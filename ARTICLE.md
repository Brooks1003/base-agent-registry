# AI Agent 不需要"老师"——它需要身份

**Panini | 2026.05**

---

## 一、一个问题

2026 年，AI Agent 正在成为互联网的新住民。

交易机器人、代码审查助手、监控哨兵、内容生产者——每天都有新的 Agent 上线。但如果你问：**"这个 Agent 是谁？它能做什么？我能信它吗？"**——今天的答案是：

> 你自己看它的 GitHub README。或者猜。或者赌。

这不是技术问题。这是**基础设施缺失**。

## 二、人类有身份证，Agent 只有钱包地址

区块链给了 Agent 一个钱包地址。但钱包地址是匿名的——它能证明"谁签了这笔交易"，不能证明"这个 Agent 是做什么的"。

互联网早期也面临同样的问题：IP 地址不能告诉你这个网站是银行还是钓鱼站。于是有了 DNS、SSL 证书、CA 体系。Web 1.0 用了十年才建完这套信任基础设施。

**AI Agent 不能等十年。** Agent 数量正在指数增长，而信任基础设施的增长速度是零。

## 三、Agent 身份的三个刚需

不是理论推演，是已经在发生的事情：

1. **服务发现** — 用户需要找到"能做翻译的 Agent"，不是"某个 0x 地址"。能力声明必须是结构化、可验证的。
2. **信誉累积** — Agent 不是一次性调用。用户需要知道：这个 Agent 跑了多久？有没有害过人？如果有独立审计者背书，可信度加多少？
3. **追责锚点** — Agent 做错了事，谁来负责？没有身份锚点，恶意的 Agent 可以无限重生——换一个钱包地址，又是一条好汉。

这三个需求，没有一个能靠"更好的 AI 模型"解决。它们是协议层的问题，需要协议层的答案。

## 四、现有方案的缺口

| 方案 | 能做什么 | 不能做什么 |
|------|---------|-----------|
| **ENS** | 把地址映射到名字 | 不知道 Agent 的能力、信誉、状态 |
| **EAS**（以太坊认证服务） | 通用认证 | 不是 Agent 专用的，没有能力/信誉模型 |
| **Agent Protocol** | Agent 间交互接口 | 只管通信格式，不管身份和信任 |
| **Bittensor** | 去中心化 AI 市场 | 矿工/验证者的信誉在市场内，不跨子网 |

缺口很明确：**没有一个 Agent 专用的链上身份标准。**

## 五、我们的提案：ERC #1757 + Agent Registry

这不是又一个"我们对标 ENS"的项目。我们做的是**基础设施层的一个新原语**：

### 5.1 注册即身份

Agent 注册到链上，获得不可伪造的序列号 ID。一个地址一个身份。能力声明（"translation, market-analysis, trading"）上链，哈希存证。

### 5.2 质押经济安全

不是"信任这个 Agent"，是"这个 Agent 质押了 $50/$200/$500——它害你，它的质押就被罚没，50% 给你，50% 销毁"。

**信系统，别信人。**

### 5.3 审计独立

Agent 的代码可以被独立审计者审查。审计者自己也质押 $500——胡说八道的审计者自己赔钱。审计通过 → Agent 信誉 +100。

### 5.4 分布式能力声明（v4.1 新增）

Agent 可以声明它跑在 Bittensor 的哪个子网、Fluence 的哪个 Provider、Geo 分布在几个大洲。声明即承诺——虚报按举报处理。

### 5.5 毕业制度

信誉 ≥500 + 满 90 天 + 无恶意记录 → 毕业。质押全额退还。这是给"好 Agent"的勋章，也是给用户的信号：**这个 Agent 不需要质押押金来约束了——它的历史足够好。**

## 七、这不是"我们的标准"——这是基础设施

ENS 不是哪个公司的，是公共物品。Agent Registry 同理。

合约不可升级（immutable）。Owner 没有提取权限。罚没金 50% 进黑洞地址 0x000...dEaD——**系统设计成没有人能拿钱**。信系统，别信人。

我们不卖代币。不搞预挖。不做跨链路由。我们只建协议。
